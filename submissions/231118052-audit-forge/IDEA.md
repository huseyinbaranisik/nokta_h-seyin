# IDEA

Track A is useful here because it tests whether a bug-reporting primitive can stay boring in the best sense: one mount point, one host boundary, and no navigation or storage leakage into the widget. The app gives the widget realistic screens, but the app still works as a plain Expo Router project if the widget disappears.

The more interesting product lesson is that small, agent-readable reports are enough to create a repair loop when the visual context is trustworthy. The customer does not need to write a ticket taxonomy; a burn-in screenshot, a screen name, and a short note already form a compact repair input.
